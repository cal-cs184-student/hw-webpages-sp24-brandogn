Overview

This homework was an exploration of 2d bezier curves and 3d bezier surfaces as well as triangle mesh representations for 3d graphics.

In part 1, we implemented bezier curves, which model curves by linearly interpolating a set of control points repeatedly. By doing this at many scales of linear interpolation, we can effectively model a curve.

In part 2, we extended this concept to bezier surfaces, using a mesh of control points. These points were linearly interpolated across one axis, and then the resulting bezier curves were interpolated along the other, resulting in a curved surface.

In part 3, we use the half-edge data structure to implement Phong shading. We did this, by calculating cross products and areas for the faces neighboring each vertex. This process was achieved by utilizing the traversal properties of half-edges to access vertices.

In part 4, we implemented edge flipping, taking inspiration by the guide to edge operations on the half-edge data structures from the cmu cs-15-462 class. Essentially, we recalculated the appropriate attributes for every half-edge, and then assign all other objects their corresponding half-edges. Some of these operation were redundant, but were computed for the sake of completeness.

In part 5, we implemented edge splitting. this process was similar to part 4, with the addition of adding new edges, faces and vertices to our triangle mesh. In doing so, we also added new half-edges, and tagged corresponding objects as isNew=true. This ensures that subdivision can work effectively later on.

In part 6, we subdivide across our entire mesh. This process requires a series of For loops. The first updates the newPosition attribute of original vertices based on neighboring positions. The second repeats this process for each edge, calculating the position of the corresponding vertex that will be added to the center of each edge. The third loop splits each edge using function from part 5. Notably, we only split edges that are not new, and edges that do not connect an old vertex to a new vertex. This ensures we do not re-split new edges, or edges formed from cutting our original edges in half. These would result in an infinite loop. The fourth loop flips any edges that connect an old to a new vertex. We use our function from part 4 here. Our final loop copies each of our vertex newPositions into Position, ensuring that vertices are adjusted to their appropriate position in the new mesh. This completes every step of the process.

The largest problem we faced was preventing incorrect edges from being re-divided during in our subdivision function. When splitting our original edges, we tagged both resulting edges as isNew=False. However, this initially resulted in repeatedly splitting these edges infinitely. To prevent this, we added the condition about not splitting edges connecting old and new vertices.

Part 1

The de Casteljau’s algorithm essentially calculates a single point of a bezier curve by a collection of control points for some ratio t∈(0,1)t \in (0, 1)t∈(0,1)﻿, where we can complete the Bezier curve by evaluating the points for all ttt﻿. It works by recursively interpolating between the control points until we only have one point left. At each recursive step, we linearly interpolate between adjacent control points based on the ratio ttt﻿, and we take the new set of points and recurse on them until we have only one point left.

Part 2

The de Casteljau algorithm extends to Bezier surfaces defined by a n×nn \times nn×n﻿ grid of control points by adding another layer of interpolation. Instead of using one ratio ttt﻿ to define a point on a Bezier curve, we use two ratios (u,v)(u,v)(u,v)﻿ to define a point on the Bezier surface. We can use the original Bezier curve algorithm to evaluate all the curves for each nnn﻿ rows with the ratio uuu﻿ to get nnn﻿ points, then use the new points as control points and use the same Bezier curve algorithm with ratio vvv﻿ to get the final point defined at (u,v)(u,v)(u,v)﻿. We can then do this for all (u,v)(u,v)(u,v)﻿ to get the full Bezier surface.

Part 3

The idea of area-weighted vertex normals is getting the normal of a given vertex by computing the normals of the neighboring faces weighted by area. We can do this by iterating through all the neighboring vertices, and getting the 3 vertices for each of our neighboring faces and corresponding vectors (while also checking for bound edge cases). We calculate the vector corresponding to the ‘sides’ of our triangle. We then compute the cross product between these to find the normal vector with the area as the magnitude. We add up all these cross products, and return the normalized sum. This value is then utilized in order to perform Phong Shading on our teapot, with the improved rendering results seen below.

Part 4

In order to perform edge flipping, we first accessed every single object in the 2-triangle section of mesh we are flipping, including half-edges, vertices, edges and faces. If one of the faces is a boundary, then we cannot perform an edge flip on it, and so we therefore do nothing. Otherwise, we rearrange every single one of our half-edges, vertices edges and faces according to the changes described in the provided guide for implementing mesh operations. We reassign every single object and its attributes according to these steps, regardless of if they are modified or not. This ensures we don’t accidentally miss any necessary reassignments.

Part 6

We followed the framework provided in the function inline comments. We implemented the given steps by creating five loops:

  • The first two were used to pre-computing the new locations of all the vertices (including the existing vertices and the new vertices on each edge)
  • The third was used to split all the original edges.
  • The fourth was used to flip all the new edges that connected a new vert to an old vert.
  • The fifth was used to move all the vertices to their new positions.

We also modified the split function to mark the new edges and vertices as new.

One small debugging trick that helped was printing out information about the edges that were being iterated through, and we also commented out certain loops to see the effect they had on the mesh (this helped with figuring out what was wrong with the positioning). We also realized that types matter (1 / 3 might not be the same as 1.0 / 3.0)

Asymmetrical Cube Subdivision

When the cube is in its normal state, we see the asymmetry forming below. However, when we pre-process by first splitting every edge on each face into two, then our cube instead subdivides to be symmetric. When our triangles are symmetrical, then our resulting subdivided object is also symmetrical.

Even though the cube looks symmetric, the defining lines that form the cube are not symmetric, and so when you subdivide along those edges, it will smooth according to the asymmetric shape formed by the lines. In order to reduce to a symmetric object, all lines in the initial object need to be symmetric.
